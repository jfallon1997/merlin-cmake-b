23 // class ComponentTracker
24 //
25 // ComponentTracker provides the primary interface for tracking
26 // operations through a series of AcceleratorComponents. An object of
27 // class ComponentTracker can be thought of as a collection of
28 // Integrator objects, each of which is responsible for
29 // "tracking" some beam-like representation through a
30 // specific component. Selection of the correct Integrator
31 // is performed via the callback mechanism SelectIntegrator(),
32 // which is called directly by Accelerator
33 // AcceleratorComponent::PrepareTracker.
34 // Concrete Tracker objects
35 // should supply the (concrete) Integrators, which when
36 // identified, are then passed back down to the concrete
37 // class via a call the virtual function SetCurrent
38 // Integrator.
39 //
40 // The callback mechanism used by Tracker replaces the
41 // canonical virtual function mechanism. It is a form of
42 // the visitor pattern, but without the circular dependency
43 // which is normally associated with that pattern. Thus
44 // the accelerator model representation can be easily
45 // extended without the need to add new visitor functions.
46
